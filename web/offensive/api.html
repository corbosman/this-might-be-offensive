TMBO API Specification

This document describes functions and their arguments that can be used to request data from the TMBO server at thismight.be.

Authentication

I've been slowly settling on authentication issues, and I believe the best method is to replicate to some degree the authentication of the site. Logging in is officially done by the login function, which will return the results of a getuser() call on success as well as set a session cookie. If you are not authenticated, you can log in on the fly by including your username and password as HTTP AUTH parameters. Due to the risks of having a password sent so often in the clear, the API will only be accessible over SSL, and will return an HTTP 301 pointing to https if an attempt is made to use it via plain HTTP.

Conventions

All string parameters are UTF-8 encoded.

Return values are determined by the filetype extension of the request. Currently only .json and .plist are explicitly accepted. XML is planned, but not implemented.

Unless otherwise specified, all functions accept arguments over both GET and POST simultaneously. All functions that deal with uploading data accept POST only.

Functions that do output filtering based on some number of parameters will AND all the parameters. A call to getuploads(type=image,userid=151) will return image uploads by themaxx, not all uploads that are either images or by themaxx.

If a returned value is a date, and your return type is plist, the date will be returned as an ISO 8601 formatted date, as per the plist spec. All other return types will return the date in the same format as it exists in the database.

Status Codes and Return Types

The following error codes will be used by the site as failure return values:

  • 200 OK - Call completed as dialed.
  • 304 Not Modified - If a Conditional GET is used, this indicates that there are no changes to report.
  • 400 Bad Request - The call is erroneous. The body will contain debugging information.
  • 401 Unauthorized - Login failure. Either your credentials weren't valid, or you did not provide any.
  • 403 Forbidden - Refusal. You have attempted to log in too many times in a rolling 30 minute period and you are blocked from any more attempts. Wait and try again later.
  • 404 Not Found - The function being requested does not exist.
  • 500 Internal Server Error - kaboom. Please report if reproducible.
  • 502 Bad Gateway - TMBO is down for maintenance. Please wait a few minutes and try your call again.

The following return types are supported:

  • plist - property list-formatted XML data
  • json - JavaScript Object Notation-formatted string
  • php - the result of the return value being PHP serialized
  • xml - XML-formatted data

XXX: 500 and 502 are not yet implemented. they will be at 1.0.

Don't be retarded

Like the RSS feeds, requests are limited to a maximum of 200 returned elements. If you want to download the entire history of the world since the dawn of time for your research project, that's fine but please sleep between requests.

Some arguments don't make sense to mix and match since filtering arguments are ANDed together. Like setting a minimum higher than a maximum. Just don't.

Because this API is new, you may find some queries take a while to complete. If you notice calls taking more than two seconds, please email numist@numist.net with the API command you are making, and try to reduce your usage until the problem is addressed.

Bits of Neat

In the future, it's reasonable to expect that the application that the user is using to browse tmbo might be neat to know, and as such, the site will use the user agent string to determine if the user is using a browser or some other method.
If you are developing an application and want it to be recognized, please send me an email with your application name and a sample user agent.

Date arguments sent in the same format as the database ("Y-m-d H:i:s" according to php date()) will not be changed and will yield the most accurate results (since there are no potential issues with time zones).

getuploads

Get one or more uploads' metadata, including (if possible) links to the uploaded files themselves, and their thumbnails. Any number of parameters can be set or unset.

Arguments:

  • type - Optional. Can be any of {image,topic,avatar}. Get uploads of the specified type.
  • userid - Optional. Integer. Get uploads by this user.
  • after - Optional. Get uploads newer than this date. Format can be seconds since the UNIX epoch or a strtotime()-parsable string.
  • before - Optional. Get uploads older than this date. Format can be seconds since the UNIX epoch or a strtotime()-parsable string.
  • max - Optional. Integer. Get uploads before this fileid, inclusive.
  • since - Optional. Integer. Get uploads since this fileid, exclusive.
  • sort - Optional. Default is date_desc. Can be any of {date_desc, date_asc, votes_asc, votes_desc}. Sorting options by votes sort by [+], disregarding [-].
  • limit - Optional. Default and maximum is 200. Format: "%d,%d" or "%d". Syntax same as LIMIT in SQL. See: http://dev.mysql.com/doc/refman/5.0/en/select.html. Depending on your arguments, fewer than limit results might get returned.

getupload

Get an uploads' metadata, including (if possible) links to the uploaded file itself, and it's thumbnail.

The data returned from this call is structurally equivalent to the data returned from getuploads, but without the array container. Because of this, a call to this function may update your pickup id.

If the upload requested is an image or topic type, the return dictionary will include a next and previous key if possible, indicating the next image or topic.

Arguments:

  • fileid - Required. Suitable for all types {image,topic,avatar}

unreadcomments

Get a list of threads with unread comments, how many comments in each, and the commentid of the oldest unread comment in each thread.

The return value of this function is equivalent to the return value of getuploads, with an extra key for each upload indicating the oldest unread comment in the thread. Note that the username value is the user that uploaded the file, and subscribed will always be 1.

Arguments:

  • sort - Optional. Default is file_asc. Can be any of {file_desc, file_asc, comment_asc, comment_desc}.
  • limit - Optional. Default and maximum is 200. Format: "%d,%d" or "%d". Syntax same as LIMIT in SQL. See: http://dev.mysql.com/doc/refman/5.0/en/select.html. Depending on your arguments, fewer than limit results might get returned.
